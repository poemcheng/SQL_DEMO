# 資料庫管理系統的歷史故事

資料庫管理系統不是一開始就長成今天這個樣子。它的歷史其實很像一部資訊科技的冒險故事：人們先被大量資料困住，接著發明方法整理資料，又因為方法太複雜而重新思考，最後一步步走到 SQL、關聯式資料庫、NoSQL 與雲端資料庫。

## 一、資料太多了：資料庫故事的開端

在電腦剛開始被企業使用的年代，資料通常是存在檔案裡。例如公司會有一個客戶檔、一個訂單檔、一個產品檔。每個系統都自己管理自己的檔案。

這種方式一開始很直覺，但問題很快出現：

- 同一個客戶資料可能在很多檔案中重複出現。
- 如果客戶改電話，每個檔案都要改。
- 不同程式使用不同格式，資料很難整合。
- 程式和資料綁得太緊，只要檔案格式改變，程式也要跟著改。

這就是資料庫管理系統誕生的背景：人們需要一個更穩定、更一致、更容易查詢的資料管理方式。

## 二、早期資料庫：階層式與網狀式模型

在關聯式資料庫出現之前，早期常見的資料庫模型是階層式資料庫與網狀式資料庫。

### 階層式資料庫

階層式資料庫像一棵樹。資料有父子關係，例如：

```text
公司
└── 部門
    └── 員工
```

IBM 的 IMS（Information Management System）就是著名的階層式資料庫。它在大型主機時代非常重要，也被用在許多金融、航空與政府系統中。

優點是結構清楚、速度快；缺點是資料關係一旦變複雜，就很難彈性查詢。

### 網狀式資料庫

網狀式資料庫比階層式更彈性，允許一筆資料有多個父節點，資料之間可以形成網路。這能表達更複雜的關係，但也讓使用者必須很清楚資料的路徑。

當時查詢資料常常像在迷宮裡走路：你要知道從哪個節點出發，經過哪些指標，最後才能找到資料。

## 三、關鍵人物：Edgar F. Codd 與關聯模型

資料庫歷史中最重要的人物之一，是 IBM 研究員 Edgar F. Codd。

1970 年，Codd 發表了關聯模型（Relational Model）的論文。他提出一個革命性想法：資料可以用「表格」來表示，而使用者不需要知道資料在硬碟上怎麼存，只要用邏輯方式描述想要的結果。

這個概念今天看起來很自然，但在當時非常大膽。

他的核心想法包括：

- 資料以表格形式表示。
- 表格由列與欄組成。
- 資料之間透過鍵值建立關係。
- 使用者應該描述「要什麼」，而不是一步步告訴電腦「怎麼找」。

這讓資料庫從「程式導向」逐漸走向「資料導向」。

## 四、一個有趣的轉折：IBM 發明了關聯模型，但 Oracle 先商業化

有趣的是，關聯模型雖然出自 IBM 研究員 Codd，但最早成功商業化關聯式資料庫的公司之一卻是 Oracle。

IBM 當時有強大的研究能力，也做出了 System R 這個重要實驗系統。System R 證明關聯式資料庫與 SQL 是可行的。

但 Oracle 看到了機會，較早推出商業產品，並積極推向市場。這成為資料庫歷史上一段常被提起的商業趣事：發明概念的人不一定最先把市場吃下來。

這也提醒我們，技術創新和產品化能力是兩件不同的事。

## 五、SQL 的誕生：讓人類可以和資料庫說話

關聯式資料庫需要一種語言來查詢資料，SQL 就是在這個背景下出現的。

SQL 原本受到 IBM System R 專案中的 SEQUEL 影響，後來因命名原因改成 SQL。它的特色是接近英文，讓使用者可以用宣告式的方式查詢資料。

例如：

```sql
SELECT customer_name, city
FROM customers
WHERE city = '台北市';
```

SQL 的重要性在於，它讓查詢資料變得比較接近「描述需求」，而不是寫一長串資料搜尋步驟。

這也是資料庫課程常講的一句話：SQL 告訴資料庫你想要什麼，不是告訴資料庫每一步要怎麼走。

## 六、主要人物與貢獻

| 人物 | 角色 | 重要貢獻 |
| --- | --- | --- |
| Edgar F. Codd | IBM 研究員 | 提出關聯模型，奠定關聯式資料庫基礎 |
| Donald D. Chamberlin | IBM 研究員 | 參與設計 SEQUEL / SQL |
| Raymond F. Boyce | IBM 研究員 | 參與 SQL 設計，Boyce-Codd Normal Form 以其命名 |
| Larry Ellison | Oracle 共同創辦人 | 推動關聯式資料庫商業化 |

## 七、資料庫設計的另一條線：正規化

當資料被放進表格後，新的問題出現了：表格要怎麼設計才不會重複、矛盾或難以維護？

這就引出了正規化（Normalization）。

正規化的目的不是讓資料表變多，而是讓資料更一致、更不容易出錯。常見概念包括：

- 每個欄位只放單一值。
- 重複資料要拆出來。
- 非主鍵欄位應該依賴主鍵。
- 多對多關係通常要用中介表處理。

例如在訂單系統中，不應該把顧客姓名、地址、電話重複存在每一筆訂單中，而是建立 `customers` 表，再讓 `orders` 透過 `customer_id` 連過去。

這就是 JOIN 與關聯式設計的核心精神。

## 八、交易處理與 ACID：資料庫要可靠

資料庫不只是拿來查詢，也常用在銀行、訂單、庫存、付款等重要系統。這些情境最怕資料做一半。

例如轉帳時：

1. A 帳戶扣款 1000 元。
2. B 帳戶加款 1000 元。

如果第一步成功、第二步失敗，就會出大問題。

因此資料庫發展出交易（Transaction）與 ACID 特性：

| 特性 | 說明 |
| --- | --- |
| Atomicity 原子性 | 交易要嘛全部成功，要嘛全部失敗 |
| Consistency 一致性 | 交易前後資料都要符合規則 |
| Isolation 隔離性 | 多個交易同時執行時不能互相干擾 |
| Durability 持久性 | 交易完成後資料要被可靠保存 |

這也是資料庫比一般檔案管理更可靠的原因之一。

## 九、從大型主機到個人電腦：資料庫走進日常工作

早期資料庫主要用在大型企業與政府系統。後來隨著個人電腦普及，資料庫逐漸走進辦公室與學校。

Microsoft Access、dBASE、FoxPro 等工具，讓許多非專業工程師也能建立簡單資料庫。雖然這些工具和大型資料庫相比規模較小，但它們對教育與企業內部管理影響很大。

許多人第一次理解資料表、表單、查詢與報表，就是從這些桌面資料庫工具開始。

## 十、網際網路時代：MySQL、PostgreSQL 與開源資料庫

網際網路興起後，網站需要大量儲存會員、商品、文章、訂單與留言資料。MySQL、PostgreSQL 等開源資料庫因此變得非常重要。

MySQL 因為容易安裝、速度快、和 PHP 搭配方便，成為許多網站的常見選擇。PostgreSQL 則以功能完整、標準支援度高、擴充能力強受到重視。

這個時期，資料庫不再只是大型企業的工具，而成為網站開發者、創業團隊與學生都能使用的基礎技術。

## 十一、NoSQL 的出現：不是不要 SQL，而是資料型態變多了

到了大型網路平台與行動應用時代，資料型態變得更複雜。除了表格資料，還有文件、社群關係、即時事件、快取資料與大量日誌。

NoSQL 資料庫因此興起，例如：

| 類型 | 例子 | 適合情境 |
| --- | --- | --- |
| 文件資料庫 | MongoDB | JSON 文件、彈性欄位 |
| Key-Value 資料庫 | Redis | 快取、即時狀態 |
| 欄族資料庫 | Cassandra | 大規模分散式寫入 |
| 圖形資料庫 | Neo4j | 社群關係、推薦、路徑分析 |

NoSQL 的出現不是代表關聯式資料庫過時，而是資料世界變大了，需要更多工具。很多現代系統會同時使用 SQL 與 NoSQL。

## 十二、雲端資料庫：資料庫變成服務

近年來，資料庫逐漸從「自己安裝與維護」變成「雲端服務」。

常見雲端資料庫服務包括 Amazon RDS、Google Cloud SQL、Azure SQL Database、BigQuery、Snowflake 等。

雲端資料庫改變了幾件事：

- 不一定要自己管理伺服器。
- 備份、擴充與監控可以交給平台。
- 資料分析可以處理更大的資料量。
- 團隊可以更快建立產品原型。

但雲端資料庫也帶來新的課題，例如成本管理、權限控管、資料安全與跨區域備份。

## 十三、資料庫歷史中的幾個趣事

### 趣事 1：SQL 的名字原本不是 SQL

SQL 的前身常被稱為 SEQUEL，意思接近 Structured English Query Language。後來因為商標問題，名稱改為 SQL。

所以有些人把 SQL 讀成「S-Q-L」，有些人讀成「sequel」。兩種讀法在業界都聽得到。

### 趣事 2：關聯模型一開始不是所有人都看好

Codd 提出關聯模型時，許多系統仍習慣用階層式或網狀式資料庫。對當時的人來說，用表格和數學關係來管理資料，看起來可能太抽象。

但後來證明，抽象反而帶來了彈性。使用者不必知道資料實際怎麼存，就能用 SQL 查出想要的結果。

### 趣事 3：資料庫常常比應用程式活得更久

很多企業系統會換介面、換程式語言、換伺服器，但資料庫可能保留很多年。因為資料才是企業真正長期累積的資產。

這也是為什麼資料庫設計很重要：一開始設計得不好，後面可能要背很久的技術債。

### 趣事 4：一個少寫的 WHERE，可能造成大災難

學 SQL 的人常聽到一個警告：`UPDATE` 和 `DELETE` 一定要小心 `WHERE`。

例如：

```sql
UPDATE employees
SET salary = 30000;
```

這會把所有員工薪水都改成 30000。真正的系統中，這種錯誤可能造成嚴重事故。因此實務上常會搭配備份、交易、權限控管與審核機制。

## 十四、技術發展脈絡總表

| 時期 | 技術重點 | 代表意義 |
| --- | --- | --- |
| 檔案處理時代 | 各系統自行管理檔案 | 簡單但容易重複與不一致 |
| 階層式 / 網狀式資料庫 | 透過路徑與指標存取資料 | 效能高但查詢彈性不足 |
| 關聯模型 | 用表格與鍵值描述資料 | 查詢邏輯與儲存方式分離 |
| SQL 標準化 | 用宣告式語言查詢資料 | 降低查詢門檻 |
| 交易與 ACID | 強調可靠性與一致性 | 支援銀行、訂單、付款等關鍵系統 |
| 開源資料庫 | MySQL、PostgreSQL 普及 | 網站與教育場景大量使用 |
| NoSQL | 文件、圖形、快取、分散式資料 | 回應新型態資料需求 |
| 雲端資料庫 | Database as a Service | 降低維運門檻，提高擴充彈性 |

## 十五、給學生的總結

資料庫管理系統的歷史，其實是在回答三個問題：

1. 資料太多，要怎麼整理？
2. 資料很重要，要怎麼保持正確？
3. 資料會被很多人使用，要怎麼讓查詢與管理更有效率？

從檔案到 SQL，從大型主機到雲端，資料庫一直在解決同一件事：讓資料成為可以被信任、被查詢、被分析、被長期保存的資產。

學 SQL 不只是學語法，而是在學一種資料思維。
